## 1. 專案概述

### 1.1 背景

將原本的紙本簽到改為線上 QR Code 簽到，目的是提升效率、減少紙本作業、同時防止代簽。本系統初期僅服務一堂課（**創新 AI 碩士班的專題討論**，這是一門多班級合開課程，但我們只負責本班的出缺席紀錄）。系統設計上**保留多課程擴充能力**，未來其他班級或其他課程要加入時，只需新增 `courses` 紀錄即可，不需動到程式碼。

### 1.2 核心目標

- **方便**：學生掃 QR + Google 一鍵登入即完成，全程無任何額外步驟
- **無感蒐證**：平常不擋任何人，但完整記錄裝置指紋，必要時助教可在後台查出代簽
- **公平**：以「掃 QR 當下時間」為打卡時間，避免登入延遲影響判定
- **可審計**：完整裝置指紋紀錄供後續追蹤
- **零維運**：全 serverless，無需自架伺服器

### 1.3 非目標（明確不做）

- 學期成績統計
- 推播提醒
- 介接學校 SSO 或學校 App API（合規與資安考量）

> 雖然初期只服務一個班級，但**多課程/多班級支援已內建在資料模型**。如果之後其他班級或老師要使用，只需要在 `courses` 表新增一筆紀錄並指定該課程的助教，不需修改程式碼。

### 2.3 業務規則

#### 上課時間判定

```
上課時間：13:10
允許簽到開始：12:40（上課前 30 分鐘）
準時範圍：12:40 ≤ T ≤ 13:10
遲到範圍：13:10 < T ≤ 13:20
缺席判定：T > 13:20 或 未簽到
```

> 「打卡時間 T」採用 **學生第一次掃 QR 並打開簽到頁的伺服器時間**，而非完成 Google 登入的時間。詳見 §4.3 打卡時間判定機制。

#### QR Code 模式

- **動態模式**：每 45 秒更新一次 nonce，需投影或平板顯示
- **靜態模式**：產生整堂課有效的 QR，貼在教室門口（無投影/無網路場景）

### 2.4 代簽防制策略（無感蒐證）

**威脅模型**：同實驗室成員之間互相代簽。

**設計哲學**：
- QR Code 只在現場投影且每 45 秒刷新，**已經能擋掉大部分隨手代簽**
- 真正想代簽的人擋不勝擋，所以**不在學生端做任何即時擋人或警示**
- 改為「無感蒐證」：完整記錄 fingerprint + IP + UA，需要時助教可在後台查
- 學生端體驗從頭到尾完全一致，沒有「可疑提示」、沒有「需要驗證」、沒有任何額外欄位

**做了什麼**：
- ✅ 每筆簽到記錄完整 FingerprintJS visitorId + raw components
- ✅ 助教控制台有「異常分析」頁，可查同裝置跨帳號、同 IP 突發等模式
- ✅ 助教可手動標記/刪除可疑紀錄

**不做什麼**：
- ❌ 不做即時擋人或警示（避免破壞體驗、避免被學生反推偵測邏輯）
- ❌ 不做口令驗證（增加摩擦）
- ❌ 不做拍照/人臉/GPS（隱私爭議大）

### 4.2 動態 QR Code 機制

**投影頁邏輯**：

```typescript
// /admin/courses/[courseId]/sessions/[id]/projector
useEffect(() => {
  const tick = async () => {
    const res = await fetch(`/api/courses/${courseId}/sessions/${id}/qr`);
    const { nonce, expires_at } = await res.json();
    setQrUrl(`https://app.xxx/scan/${nonce}`);
  };
  tick();
  const timer = setInterval(tick, 45_000);  // 每 45 秒
  return () => clearInterval(timer);
}, [courseId, id]);
```

**Worker 端產生 nonce**：

```typescript
// /api/courses/[courseId]/sessions/[id]/qr
export async function GET(req, { params }) {
  await requireCourseAdmin(req, params.courseId);  // §5.3
  
  const nonce = crypto.randomUUID();
  const created_at = Date.now();
  const expires_at = created_at + 60_000;  // 60 秒有效（給 45 秒換的緩衝）
  
  // 寫 KV：給 /api/scan 驗證用
  await env.KV.put(
    `nonce:${nonce}`,
    JSON.stringify({
      course_id: params.courseId,
      session_id: params.id,
      created_at,            // ★ 給 reaction_ms 計算用
      expires_at,
    }),
    { expirationTtl: 60 }
  );
  
  // 寫 D1 nonce_log：給秒簽偵測用（KV 過期後 D1 仍保留）
  await env.DB.prepare(
    'INSERT INTO nonce_log (nonce, session_id, created_at) VALUES (?, ?, ?)'
  ).bind(nonce, params.id, created_at).run();
  
  return Response.json({ nonce, expires_at });
}
```

**靜態 QR 模式**：
助教在控制台點擊「切換靜態 QR」時，產生一個整堂課有效的特殊 nonce（前綴 `static-`），KV key 為 `nonce:static:{nonce}`，TTL 設為 3 小時。後端驗證時同時檢查 `nonce:{nonce}` 和 `nonce:static:{nonce}` 兩個 key。

### 4.3 打卡時間判定機制（關鍵）

這是整個系統最重要的設計，避免「登入太久 → 被算遲到」的不公平。

#### Step 1: 學生掃 QR 進入落地頁

落地頁 mounted 後**立即**呼叫 `/api/scan`，不需要任何登入：

```typescript
// /scan/[nonce]/page.tsx
useEffect(() => {
  (async () => {
    const fp = await getFingerprint();
    const res = await fetch('/api/scan', {
      method: 'POST',
      body: JSON.stringify({ nonce, fingerprint: fp }),
    });
    const { pending_id } = await res.json();
    sessionStorage.setItem('pending_id', pending_id);
    // 接著觸發 Google 登入
    signIn('google', { callbackUrl: `/api/checkin?pid=${pending_id}` });
  })();
}, []);
```

#### Step 2: 伺服器記下 scan_time

```typescript
// /api/scan
export async function POST(req) {
  const { nonce, fingerprint } = await req.json();
  
  // 驗證 nonce（從 KV 取出當時記下的 metadata，包含產生時間）
  const nonceData = await env.KV.get(`nonce:${nonce}`, 'json');
  if (!nonceData) return Response.json({ error: 'invalid_nonce' }, { status: 400 });
  // nonceData 結構：{ course_id, session_id, created_at, expires_at }
  
  // 建立 pending
  const pending_id = crypto.randomUUID();
  const scan_time = Date.now();  // ★ 關鍵：以伺服器收到請求的時間為準
  
  await env.KV.put(
    `pending:${pending_id}`,
    JSON.stringify({
      course_id: nonceData.course_id,
      session_id: nonceData.session_id,
      nonce_created_at: nonceData.created_at,  // ★ 帶下去算 reaction_ms 用
      scan_time,
      fingerprint,                              // 前端傳上來的 { visitorId, components }
      ip: req.headers.get('cf-connecting-ip'),
      user_agent: req.headers.get('user-agent'),
    }),
    { expirationTtl: 600 }  // 10 分鐘
  );
  
  return Response.json({ pending_id });
}
```

> §4.2 的 `/api/session/{id}/qr` 在產生 nonce 時必須同時把 `created_at` 寫進 KV value，並且 INSERT 一筆到 `nonce_log` 表（schema 見 §5.1）。這是異常分析「秒簽偵測」的資料來源。

#### Step 3: Google 登入完成後正式打卡

```typescript
// /api/checkin
export async function GET(req) {
  const session = await auth();  // NextAuth v5 helper
  if (!session?.user?.email?.endsWith('@ntut.org.tw')) {
    return Response.redirect('/error?code=invalid_domain');
  }
  
  const pending_id = new URL(req.url).searchParams.get('pid');
  if (!pending_id) return Response.redirect('/error?code=missing_pid');
  
  // 取出 pending（不立刻刪除，等 DB 寫入成功才刪，避免 race condition）
  const pending = await env.KV.get(`pending:${pending_id}`, 'json');
  if (!pending) {
    return Response.redirect('/error?code=pending_expired');
  }
  
  // ★ 用 pending.scan_time 而不是 Date.now() 來判定狀態
  // 先從 D1 查出這堂課的時間規則
  const sessionRow = await env.DB.prepare(
    'SELECT early_open_at, class_start_at, late_cutoff_at FROM sessions WHERE id = ?'
  ).bind(pending.session_id).first();
  if (!sessionRow) return Response.redirect('/error?code=invalid_session');
  
  const status = determineStatus(
    pending.scan_time,
    sessionRow.early_open_at,
    sessionRow.class_start_at,
    sessionRow.late_cutoff_at,
  );
  const reaction_ms = pending.scan_time - pending.nonce_created_at;
  
  // 寫入 attendance（欄位順序與 §5.1 schema 完全對齊）
  try {
    await env.DB.prepare(`
      INSERT INTO attendance 
        (session_id, course_id, user_email, user_name,
         scan_time, login_time, status,
         fingerprint_hash, fingerprint_raw,
         ip, user_agent, reaction_ms, created_at)
      VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
    `).bind(
      pending.session_id,
      pending.course_id,
      session.user.email,
      session.user.name ?? null,
      pending.scan_time,
      Date.now(),
      status,
      pending.fingerprint?.visitorId ?? null,
      pending.fingerprint?.components ? JSON.stringify(pending.fingerprint.components) : null,
      pending.ip,
      pending.user_agent,
      reaction_ms,
      Date.now(),
    ).run();
  } catch (err: any) {
    // UNIQUE(session_id, user_email) 撞到 → 學生重複觸發，視為已完成
    if (String(err?.message ?? err).includes('UNIQUE')) {
      await env.KV.delete(`pending:${pending_id}`);
      return Response.redirect(`/result?status=already_signed`);
    }
    throw err;
  }
  
  // ★ 寫入成功後才刪 pending（DB 才是 source of truth）
  await env.KV.delete(`pending:${pending_id}`);
  return Response.redirect(`/result?status=${status}`);
}
```

**Race condition 處理說明**：

KV.get → DB.insert → KV.delete 三步本質上不是原子操作。可能的競態：

1. **OAuth callback 重試**：瀏覽器 redirect 重複觸發 → 第二次 INSERT 撞到 `UNIQUE(session_id, user_email)` → catch 捕捉 → 回傳 `already_signed`，使用者體驗一致
2. **DB 寫入失敗**：不刪 pending → 學生可重試 → 下次成功為止
3. **DB 寫入成功但 KV.delete 失敗**：pending 殘留但 10 分鐘後自動過期；期間若使用者重試會撞 UNIQUE 進入 `already_signed` 分支，**不會產生重複紀錄**

**結論**：DB 的 UNIQUE 約束才是真正的「打卡只有一次」保證，KV pending 只是時間戳的暫存與授權通行證。

#### Step 4: 狀態判定函式

```typescript
// lib/status.ts
/**
 * 純函式：只做時間比較，不碰 DB。
 * 呼叫端（/api/checkin）負責先從 D1 查出 session 的三個時間點再傳入。
 */
export function determineStatus(
  scan_time_ms: number,
  early_open: number,    // Unix ms，例 12:40
  class_start: number,   // Unix ms，例 13:10
  late_cutoff: number,   // Unix ms，例 13:20
): 'too_early' | 'on_time' | 'late' | 'absent' {
  if (scan_time_ms < early_open) return 'too_early';
  if (scan_time_ms <= class_start) return 'on_time';
  if (scan_time_ms <= late_cutoff) return 'late';
  return 'absent';
}
```

### 4.5 Fingerprint 蒐證機制（後台無感記錄）

`/api/checkin`（見 §4.3 的權威實作）寫入 attendance 時會把 FingerprintJS 的 `visitorId`、完整 components JSON、IP、User-Agent、`reaction_ms` 一併記下，**不做任何即時判定**：

- 寫入欄位：`fingerprint_hash`（visitorId）、`fingerprint_raw`（components JSON）、`ip`、`user_agent`、`reaction_ms`
- 不做：碰撞偵測、警告、可疑標記
- 結果：學生端永遠看到「✅ 簽到成功」，所有判定移到 §4.6 異常分析儀表板由助教手動觸發

### 4.6 異常分析儀表板（助教後台查證用）

控制台 `/admin/analytics` 頁，需要時點進來查。常用查詢：

```sql
-- 1. 同 visitorId 跨帳號（最有效，找實驗室代簽）
SELECT 
  fingerprint_hash,
  COUNT(DISTINCT user_email) as account_count,
  GROUP_CONCAT(DISTINCT user_email) as accounts,
  COUNT(*) as total_signs
FROM attendance
WHERE fingerprint_hash IS NOT NULL
GROUP BY fingerprint_hash
HAVING account_count > 1
ORDER BY account_count DESC;

-- 2. 同 IP 突發（同一 Wi-Fi 短時間多人）
SELECT session_id, ip, 
       COUNT(DISTINCT user_email) as cnt,
       GROUP_CONCAT(user_email) as users,
       MIN(scan_time) as first_scan,
       MAX(scan_time) as last_scan
FROM attendance
GROUP BY session_id, ip
HAVING cnt >= 3
ORDER BY cnt DESC;

-- 3. 秒簽偵測（QR 出現後反應時間 < 5 秒，可能是預先準備好截圖）
SELECT user_email, session_id, scan_time, reaction_ms
FROM attendance
WHERE reaction_ms IS NOT NULL AND reaction_ms < 5000
ORDER BY reaction_ms ASC;

-- 4. 跨堂課累積：某 visitorId 整學期幫過幾個人
SELECT fingerprint_hash,
       COUNT(DISTINCT user_email) as helped_count,
       COUNT(DISTINCT session_id) as session_count
FROM attendance
GROUP BY fingerprint_hash
HAVING helped_count > 1
ORDER BY helped_count DESC;
```

UI 上用三個折疊區塊呈現：
- 🔍 **同裝置跨帳號**（最重要）
- 🌐 **同 IP 突發**
- ⏱ **秒簽紀錄**

每筆都可展開看 raw fingerprint components，以及「標記為已處理」「刪除此紀錄」的操作按鈕。

### 4.8 Auth.js 設定（JWT session 策略）

採 **JWT session strategy**，理由：
- Session 狀態存在加密 cookie 內，**不需要 D1 adapter 的標準四張表**（`users`, `accounts`, `sessions`, `verification_tokens`）
- Cloudflare Workers edge 環境下，少一次 DB 查詢即代表更低延遲
- 我們不需要傳統 session 表的「強制登出」功能（學期結束直接換 NEXTAUTH_SECRET 即可）

```typescript
// lib/auth.ts
import NextAuth from 'next-auth';
import Google from 'next-auth/providers/google';

export const { handlers, auth, signIn, signOut } = NextAuth({
  session: {
    strategy: 'jwt',           // ★ 關鍵：採 JWT，不用 DB adapter
    maxAge: 60 * 60 * 8,       // 8 小時
  },
  providers: [
    Google({
      clientId: process.env.GOOGLE_CLIENT_ID!,
      clientSecret: process.env.GOOGLE_CLIENT_SECRET!,
    }),
  ],
  callbacks: {
    async signIn({ user }) {
      // 限制網域
      return !!user.email?.endsWith('@ntut.org.tw');
    },
    async jwt({ token, user }) {
      // 第一次登入時把 email 寫進 token
      if (user) token.email = user.email;
      return token;
    },
    async session({ session, token }) {
      // 把 token.email 暴露到 session 物件
      if (session.user) session.user.email = token.email as string;
      return session;
    },
  },
  secret: process.env.NEXTAUTH_SECRET,
});
```

**權限檢查不在 NextAuth callback 內做**，因為 callback 內只認得「網域」這個粗粒度條件。細粒度的「是不是某課程的助教」由 §5.3 的 `requireCourseAdmin()` 在每個 API route 內檢查 DB。

> 如果未來改成 database session strategy，需要額外建 `users`, `accounts`, `sessions`, `verification_tokens` 四張 Auth.js 標準表，並安裝 `@auth/d1-adapter`。本版不採用此路線。

### 5.3 多課程的權限模型

| 角色 | 來源 | 權限範圍 |
|------|------|----------|
| 學生 | Google OAuth 通過 `@ntut.org.tw` 網域檢查 | 只能簽到，看自己的紀錄 |
| 課程助教 (`ta`) | `course_admins` 表 | 管理該課程的 sessions、紀錄、匯出 |
| 課程教師 (`instructor`) | `course_admins` 表 | 同上 + 管理該課程助教 |
| 課程擁有者 (`owner`) | `course_admins` 表 | 同上 + 封存課程 |
| 系統管理員 | `super_admins` 表 | 跨課程，可建立新課程、看所有資料 |

權限檢查邏輯（middleware）：

```typescript
import { auth } from '@/lib/auth';

async function requireCourseAdmin(req, courseId) {
  const session = await auth();  // NextAuth v5 helper
  if (!session?.user?.email) throw new Error('unauthorized');
  
  // 1. 先查 super_admins
  const superAdmin = await env.DB.prepare(
    'SELECT email FROM super_admins WHERE email = ?'
  ).bind(session.user.email).first();
  if (superAdmin) return { role: 'super', email: session.user.email };
  
  // 2. 再查該課程的 course_admins
  const courseAdmin = await env.DB.prepare(
    'SELECT role FROM course_admins WHERE course_id = ? AND email = ?'
  ).bind(courseId, session.user.email).first();
  if (courseAdmin) return { role: courseAdmin.role, email: session.user.email };
  
  throw new Error('forbidden');
}
```

### 5.4 KV 結構

| Key 模式 | Value | TTL | 用途 |
|----------|-------|-----|------|
| `nonce:{uuid}` | `{ course_id, session_id, created_at, expires_at }` | 60s | 動態 QR 的當前有效 nonce |
| `nonce:static:{uuid}` | `{ course_id, session_id, created_at, expires_at }` | 課程時長 | 靜態 QR |
| `pending:{uuid}` | `{ course_id, session_id, nonce_created_at, scan_time, fingerprint, ip, user_agent }` | 600s | 掃了但還沒登入的暫存 |

> `nonce_created_at` 從 nonce KV 帶到 pending KV，最終寫入 `attendance.reaction_ms = scan_time - nonce_created_at`，作為「秒簽偵測」的資料來源。

## 6. 安全性設計

### 6.1 認證與授權

| 角色 | 認證方式 | 授權檢查 |
|------|----------|----------|
| 學生 | Google OAuth (JWT session) | NextAuth `signIn` callback 檢查 email 結尾為 `@ntut.org.tw` |
| 課程助教 | Google OAuth (JWT session) | API route 內 `requireCourseAdmin(req, courseId)` 查 `course_admins` 表 |
| 系統管理員 | Google OAuth (JWT session) | 同上，先查 `super_admins` 表，命中即放行所有課程 |

> **權限來源唯一性**：所有助教/管理員權限**只能**從 `course_admins` 與 `super_admins` 兩張 DB 表查得，不存在環境變數白名單、不存在程式碼硬編碼。新增/移除助教都透過 DB 操作或控制台 UI 完成。

> **學生簽到不檢查修課名單**：這是刻意設計（見 §5.1 `enrolled_students` 表的註解）。任何 `@ntut.org.tw` 帳號只要掃到有效 QR 都能完成簽到，非本班學生由助教課後在後台處理。

### 6.2 防代簽機制

| 層 | 機制 | 防什麼 | 類型 |
|----|------|--------|------|
| L1 | 動態 QR（45 秒）| 截圖事後傳給沒到的人 | 即時擋 |
| L2 | 帳號綁定（`@ntut.org.tw`）| 用陌生人帳號簽 | 即時擋 |
| L3 | UNIQUE(session, email) | 重複簽到 | 即時擋 |
| L4 | Fingerprint 完整紀錄 | 同裝置跨帳號代簽 | 後台蒐證 |
| L5 | IP + UA 紀錄 | 同 Wi-Fi 連坐 | 後台蒐證 |
| L6 | 反應時間紀錄 | 預先準備好截圖 | 後台蒐證 |
| L7 | 異常分析儀表板 | 跨堂課累積查證 | 後台蒐證 |
| L8 | 助教手動處理 | 確認後刪除/修改紀錄 | 人工 |

> **設計原則**：L1-L3 是即時擋人的硬規則，L4-L8 是「無感蒐證」——學生端從掃 QR 到看到「✅ 簽到成功」全程一致，沒有任何可疑提示或額外驗證步驟。所有判定都由助教在後台進行。

### 6.3 隱私與資料保護

- 所有資料存在 Cloudflare 亞太節點
- Fingerprint 採 hash + raw 雙存，必要時可只刪 raw
- 學期結束後可一鍵清除所有 attendance 紀錄
- 不蒐集 GPS 定位（避免合規問題）
- Privacy notice 顯示在首次登入頁

### 7.2 Google OAuth 設定

1. 進 https://console.cloud.google.com 建專案
2. OAuth consent screen → External → 填寫資訊
3. Scopes 只勾 `openid`, `email`, `profile`
4. Credentials → Create OAuth Client ID → Web application
5. Authorized redirect URIs：
   - `https://attendance-app.<your-account>.workers.dev/api/auth/callback/google`（Workers 預設網域）
   - 或自訂網域 `https://attendance.your-domain.com/api/auth/callback/google`
   - `http://localhost:3000/api/auth/callback/google`（本地開發）
6. Publish App（直接切 Production，不需 Google 審核）

### 7.3 自訂網域（可選）

Cloudflare Dashboard → Workers & Pages → 選擇 `attendance-app` → Settings → Domains & Routes → 加 `attendance.your-domain.com`。如果網域 DNS 已託管在 Cloudflare，會自動完成 DNS 設定。

### 7.4 監控

- Cloudflare Analytics：免費的 request、error rate 監控
- D1 Console：直接在 Cloudflare Dashboard 看資料
- Worker logs：`pnpm wrangler tail` 即時看 log
- Workers Observability：Dashboard 的 Workers Metrics 頁可看延遲、CPU 時間等

## 8. 邊界情況與處理

| 情況 | 處理方式 |
|------|----------|
| 學生掃 QR 後不登入 | KV pending 10 分鐘後自動過期 |
| 學生 nonce 過期 | 顯示「QR 已失效，請重新掃描」 |
| 學生不是 `@ntut.org.tw` 帳號 | sign-in callback 回 false，導向錯誤頁 |
| 同學生連掃兩次 | 第二次 INSERT 會撞 UNIQUE，回傳「已簽到」 |
| 教室斷網 | 切換靜態模式，學生用自己 4G 簽到 |
| 助教忘記開 session | 控制台顯示「今天還沒開啟簽到」紅字提醒 |
| 助教關閉 session 後有人簽 | session.status='closed' 時所有 nonce 失效 |
| 學生手機相機壞了 | 助教用控制台手動補簽 |
| 學生忘記帶手機 | 同上 |
| 系統當機 | 退回紙本，事後助教手動補登 |
